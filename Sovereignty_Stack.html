The Sovereignty Stack

A complete guide to running Bitcoin Core, Fulcrum, and Sparrow Wallet on a single dedicated Debian 13 laptop — with full-disk encryption, XFCE desktop, Bitcoin Core peer privacy via Tor, and no third-party trust.

Bitcoin Core v30.2
Fulcrum v2.1.0
Sparrow v2.2.0
OS Debian 13 (Trixie)
Storage Required ≥ 2 TB SSD
00 — Overview

Architecture & Data Flow

This guide builds a fully self-sovereign Bitcoin infrastructure for personal use. Understanding the data flow before you begin will prevent confusion later:

Architecture
  ┌──────────────────────────────────────────────────────────────┐
  │  Single Laptop (Debian 13, LUKS-encrypted, XFCE)            │

Time-Gates Summary

There are two mandatory waiting periods in this process. Both are hardware and network-speed dependent. Nothing can be done to skip them.

⏱
Time-Gate 1 — Initial Block Download (IBD)
Bitcoin Core must download and validate the entire blockchain (~700 GB as of 2025) before Fulcrum can begin indexing. On a mid-range machine with a fast SSD and good internet connection, expect 2–5 days. This cannot be skipped or parallelized. You can configure Fulcrum before this completes, but it will not start indexing until bitcoind reports "initialblockdownload": false.
⏱
Time-Gate 2 — Fulcrum Index Build
After IBD completes, Fulcrum builds its own address index from the full blockchain. This takes an additional 4–20 hours depending on RAM allocated and drive speed. Fulcrum will not accept Sparrow connections until this index is complete — connection attempts will be refused.
Total Minimum Time
Plan for 3–7 days from a fresh Debian install to a fully operational stack. The machine does not need to be supervised during this time; both services run as systemd daemons and survive reboots. XFCE is installed after both time-gates complete — not during OS install — so it does not interfere with resource-intensive sync operations.
01 — Preparation

Storage Planning

Adequate storage is the single most common failure point in this setup. Plan your disk layout before installing the OS.

Component Size (2025) Growth/yr Notes
Bitcoin blockchain ~700 GB ~60 GB Full archival, no pruning
Fulcrum index ~120 GB ~12 GB RocksDB, must be SSD
OS + software ~20 GB minimal /boot unencrypted (~1 GB)
Swap 8–16 GB — Critical for Fulcrum indexing
Total minimum ~870 GB ~75 GB Use ≥ 2 TB SSD for headroom
SSD Required for Fulcrum
Fulcrum's RocksDB index performs millions of random I/O operations during sync. On a spinning HDD, the index build will take weeks and may appear hung. An SSD is not optional for the data partition.

Debian 13 (Trixie) Installation

Download the official Debian 13 netinstall ISO. Verify it before writing to USB.

Disk Partitioning with Full-Disk Encryption

During the Debian installer partitioning step, select "Guided — use entire disk and set up encrypted LVM". This is the recommended path for a dedicated machine. What follows are the specific choices that matter for this setup:

1
Choose partitioning scheme

When prompted for the partitioning scheme, choose "All files in one partition". We will be placing the Bitcoin data directory on this same encrypted volume. A separate data partition is not necessary when using LVM since you can extend volumes later if needed.

2
Encryption passphrase

Choose a strong passphrase — this is the master key to everything on this machine. 6+ random words (diceware) is recommended. Write it down and store it securely offline. There is no recovery mechanism if you forget this.

3
Amount of volume group to use

Use 100% of the volume group. We need all available space for the blockchain.

4
Resulting partition layout

The installer will create: a small unencrypted /boot partition (~1 GB), a LUKS-encrypted container, and inside it an LVM volume group containing / (root) and swap. This is the correct layout.

Why Encrypt the Swap?
The Debian installer will warn you if swap is unencrypted while using an encrypted LVM. Accept its recommendation to encrypt swap as well. The LUKS key is held in memory and can potentially be written to an unencrypted swap partition, defeating the purpose of disk encryption.
Expected layout after installation
NAME                     SIZE   TYPE  MOUNTPOINT
sda                      2TB    disk
├─sda1                   1GB    part  /boot          # unencrypted
└─sda2                   ~2TB   part
  └─sda2_crypt           ~2TB   crypt               # LUKS container
    ├─debian--vg-root    ~1.9TB lvm   /
    └─debian--vg-swap    16GB   lvm   [SWAP]

Software Selection

When the installer asks which software to install, deselect everything except:

  • SSH server — required for administration during the headless sync phase
  • Standard system utilities — keep this checked
Do not install a desktop environment here
XFCE is added later, after Bitcoin Core IBD and Fulcrum indexing are both complete. Installing a DE now would consume RAM that IBD and Fulcrum need during their resource-intensive phases. The machine runs headless for the first several days — this is intentional.

Post-Install Hardening

After first boot, log in via SSH (or directly) and apply the following. Run all commands as root or with sudo.

Update the system

bash
apt update && apt upgrade -y

Install essential packages

bash
apt install -y \
  ufw \
  fail2ban \
  unattended-upgrades \
  apt-listchanges \
  gnupg \
  wget \
  curl \
  git \
  htop \
  lm-sensors

Create a dedicated service user

Never run Bitcoin Core or Fulcrum as root. Create an unprivileged user for each service.

bash
# Bitcoin Core service user (no login shell, no home login)
useradd -r -m -d /var/lib/bitcoin -s /bin/bash bitcoin

# Fulcrum service user
useradd -r -m -d /var/lib/fulcrum -s /bin/bash fulcrum

Configure UFW firewall

bash
# Allow SSH (adjust port if you changed it)
ufw allow 22/tcp

# Allow Bitcoin Core P2P (optional — only if you want

# ufw allow 8333/tcp

# Fulcrum listens only on loopback — Tor handles external access
# No UFW rule needed for Fulcrum ports

ufw enable
ufw status verbose

Enable automatic security updates

bash — /etc/apt/apt.conf.d/20auto-upgrades
dpkg-reconfigure --priority=low unattended-upgrades
# Select "Yes" when prompted

Harden SSH

Edit /etc/ssh/sshd_config and ensure these values are set:

/etc/ssh/sshd_config

Ensure your SSH public key is already in ~/.ssh/authorized_keys before setting PasswordAuthentication no. Locking yourself out requires physical access to type the LUKS passphrase and fix the config.

Configure fail2ban

bash
systemctl enable fail2ban
systemctl start fail2ban
# Default config will protect SSH automatically
02 — Bitcoin Core

Download & Verify Bitcoin Core 30.2

Bitcoin Core uses a threshold signature model as of v22.0 — multiple developers independently sign the SHA256SUMS file. You verify that several keys you trust have all signed the same checksums.

1
Download the binary and checksum files
bash
cd /tmp
VERSION=30.2
BASE_URL=https://bitcoincore.org/bin/bitcoin-core-${VERSION}

wget ${BASE_URL}/bitcoin-${VERSION}-x86_64-linux-gnu.tar.gz
wget ${BASE_URL}/SHA256SUMS
wget ${BASE_URL}/SHA256SUMS.asc
2
Import developer signing keys

Clone the guix.sigs repository which contains all current builder public keys, then import them into GPG.

bash
git clone https://github.com/bitcoin-core/guix.sigs /tmp/guix.sigs
gpg --import /tmp/guix.sigs/builder-keys/*
3
Verify the SHA256SUMS file is signed by trusted developers
bash
gpg --verify SHA256SUMS.asc SHA256SUMS

# You should see multiple "Good signature" lines, e.g.:
# gpg: Good signature from "Ava Chow <...>"
# gpg: Good signature from "fanquake <...>"
# Some "WARNING: This key is not certified" messages are expected
# if you haven't personally verified the developers' key fingerprints.
# What matters: multiple independent Good signatures are present.
4
Verify the binary checksum
bash
sha256sum --ignore-missing --check SHA256SUMS
# Expected output:
# bitcoin-30.2-x86_64-linux-gnu.tar.gz: OK

If the output says anything other than OK, do not proceed. Delete all downloaded files and retry from a different network.

Install Bitcoin Core

bash
# Extract the tarball
tar -xzf /tmp/bitcoin-30.2-x86_64-linux-gnu.tar.gz -C /tmp

# Install binaries to /usr/local/bin
install -m 0755 -o root -g root \
  /tmp/bitcoin-30.2/bin/bitcoind \
  /tmp/bitcoin-30.2/bin/bitcoin-cli \
  /usr/local/bin/

# Verify installation
bitcoind --version
# Bitcoin Core version v30.2.0

# Create data directory with correct ownership
mkdir -p /var/lib/bitcoin
chown -R bitcoin:bitcoin /var/lib/bitcoin

Configure Bitcoin Core

Create the configuration file. Key decisions here: we enable txindex=1 (required by Fulcrum), and we restrict RPC access to localhost only.

bash
mkdir -p /var/lib/bitcoin/.bitcoin
nano /var/lib/bitcoin/.bitcoin/bitcoin.conf
/var/lib/bitcoin/.bitcoin/bitcoin.conf
# ─── Network ────────────────────────────────────
server=1                  # enable JSON-RPC server
listen=1                  # accept incoming connections
txindex=1                 # REQUIRED for Fulcrum

# ─── RPC ────────────────────────────────────────
rpcuser=bitcoinrpc
rpcpassword=CHANGE_THIS_TO_A_LONG_RANDOM_STRING
rpcbind=127.0.0.1
rpcallowip=127.0.0.1

# ─── Performance ────────────────────────────────
dbcache=4096              # 4 GB RAM for IBD (reduce after sync)
maxmempool=300
maxconnections=40

# ─── Tor (hybrid — outbound via Tor after IBD) ──
# Uncomment these AFTER IBD is complete
# proxy=127.0.0.1:9050
# onlynet=onion
# addnode=... (add some .onion seed nodes)

# ─── Paths ──────────────────────────────────────
datadir=/var/lib/bitcoin/.bitcoin

# ─── Logging (reduce noise) ─────────────────────
printtoconsole=0
shrinkdebugfile=1
Change the RPC password
Replace CHANGE_THIS_TO_A_LONG_RANDOM_STRING with a randomly generated string. You can generate one with: openssl rand -base64 32. Record it — you will need it for the Fulcrum config.
dbcache note
The dbcache=4096 setting (4 GB) accelerates IBD significantly on a machine with 16+ GB RAM. After IBD is complete, you should reduce this to 512 or 1024 to free memory for Fulcrum. Edit the config and restart bitcoind.
bash — set correct ownership
chown -R bitcoin:bitcoin /var/lib/bitcoin/.bitcoin
chmod 600 /var/lib/bitcoin/.bitcoin/bitcoin.conf

Create Bitcoin Core Systemd Service

/etc/systemd/system/bitcoind.service
[Unit]
Description=Bitcoin Core Node
After=network-online.target
Wants=network-online.target

[Service]
User=bitcoin
Group=bitcoin
Type=forking
ExecStart=/usr/local/bin/bitcoind \
    -conf=/var/lib/bitcoin/.bitcoin/bitcoin.conf \
    -daemon
ExecStop=/usr/local/bin/bitcoin-cli \
    -conf=/var/lib/bitcoin/.bitcoin/bitcoin.conf \
    stop
Restart=on-failure
RestartSec=30
TimeoutStopSec=300

# Security hardening
PrivateTmp=true
ProtectSystem=full
NoNewPrivileges=true
PrivateDevices=true

[Install]
WantedBy=multi-user.target
bash
systemctl daemon-reload
systemctl enable bitcoind
systemctl start bitcoind

# Verify it started correctly
systemctl status bitcoind
journalctl -u bitcoind -f

Initial Block Download

⏱
Time-Gate 1 — Active Now
IBD has started. The blockchain sync will run in the background. You can monitor progress and continue configuring Tor during this period. Do not start Fulcrum, install XFCE, or install Sparrow until IBD is confirmed complete.

Monitor IBD progress

bash — run as user bitcoin or with sudo -u bitcoin
# Check sync status (run periodically)
sudo -u bitcoin bitcoin-cli \
  -conf=/var/lib/bitcoin/.bitcoin/bitcoin.conf \
  getblockchaininfo | grep -E "blocks|headers|verificationprogress|initialblockdownload"

# You are done when:
#   "initialblockdownload": false
#   "verificationprogress": 0.9999... (very close to 1.0)

Check disk usage during sync

bash
df -h /var/lib/bitcoin
du -sh /var/lib/bitcoin/.bitcoin/blocks
Post-IBD: Reduce dbcache
Once "initialblockdownload": false, edit bitcoin.conf and change dbcache=4096 to dbcache=512. Then restart: systemctl restart bitcoind. This frees 3.5 GB of RAM for Fulcrum's indexing phase.
03 — Tor

Install Tor

Since Sparrow and Fulcrum run on the same machine, there is no need to expose Fulcrum as a hidden service — Sparrow connects directly over loopback. Tor is still valuable here for one important purpose: routing Bitcoin Core's outbound peer connections through Tor after IBD completes, which prevents your IP address from being associated with your node by the peers you connect to.

What Tor does and doesn't protect here
With this single-machine setup, your wallet queries never leave the machine at all — Sparrow talks to Fulcrum over 127.0.0.1. The remaining privacy exposure is at the Bitcoin peer-to-peer layer: without Tor, your ISP and the peers your node connects to can see your IP address and correlate it with node activity. Routing Bitcoin Core's peer connections through Tor closes that gap.
bash
apt install -y tor

# Verify Tor is running
systemctl status tor
tor --version

No additional torrc configuration is needed at this stage. Tor's default SOCKS5 proxy on 127.0.0.1:9050 is all we need.

Bitcoin Core Tor Peer Configuration (Post-IBD)

This configuration is applied after IBD is complete. Do not enable it during initial sync — IBD on clearnet is significantly faster and this setting is irrelevant to the sync process itself.

Apply after IBD only
Setting proxy=127.0.0.1:9050 during IBD will route the entire blockchain download through Tor. This would extend IBD from days to potentially weeks. Configure this step only once "initialblockdownload": false.

Once IBD is confirmed complete, add the bitcoin user to the tor group and update the Bitcoin Core config:

bash
# Allow the bitcoin service user to access Tor's socket
usermod -aG debian-tor bitcoin

Edit /var/lib/bitcoin/.bitcoin/bitcoin.conf and uncomment or add the Tor proxy lines:

/var/lib/bitcoin/.bitcoin/bitcoin.conf — add post-IBD
# ─── Tor peer routing (add after IBD is complete) ───
# Routes all outbound peer connections through Tor.
# Your IP is not exposed to any peer you connect to.
proxy=127.0.0.1:9050

# Recommended: allow both clearnet and .onion peers.
# This gives good peer diversity and is more reliable than onion-only.
# If you want strict onion-only, add: onlynet=onion
# (but expect fewer peers and occasional connection difficulty)
bash — restart and verify
systemctl restart bitcoind

# After a few minutes, check that Tor peers appear
sudo -u bitcoin bitcoin-cli \
  -conf=/var/lib/bitcoin/.bitcoin/bitcoin.conf \
  getpeerinfo | grep '"network"'
# You should see a mix of "onion" and "ipv4"/"ipv6" network entries
# With onlynet=onion set, all entries will show "onion"

Fulcrum Configuration File

Since Sparrow connects to Fulcrum locally over loopback, no SSL certificate is needed — the connection never leaves the machine. Plain TCP on 127.0.0.1 is appropriate and simpler.

/var/lib/fulcrum/fulcrum.conf
# ─── Bitcoin Core RPC connection ─────────────────────
bitcoind = 127.0.0.1:8332
rpcuser = bitcoinrpc
rpcpassword = SAME_PASSWORD_AS_BITCOIN_CONF

# ─── Data directory ──────────────────────────────────
datadir = /var/lib/fulcrum/db

# ─── Listen on loopback only ─────────────────────────
# Sparrow connects locally — no external exposure needed
tcp = 127.0.0.1:50001

# ─── Performance tuning ──────────────────────────────
# Adjust db_mem based on available RAM (in MB)
# Leave ~4 GB for OS + bitcoind, give rest to Fulcrum during indexing
# For 16 GB system: db_mem = 10000 during indexing, reduce to 4000 after
db_mem = 4000

# ─── Server identity (optional, for display) ─────────
banner = Personal Fulcrum Server

# ─── Peering (disable — not a public server) ─────────
announce_to_peers = false
peer_discovery = false
bash — set permissions
chown fulcrum:fulcrum /var/lib/fulcrum/fulcrum.conf
chmod 600 /var/lib/fulcrum/fulcrum.conf
mkdir -p /var/lib/fulcrum/db
chown -R fulcrum:fulcrum /var/lib/fulcrum/db

Fulcrum Systemd Service

Fulcrum must start after bitcoind is fully up. The After= and Requires= directives enforce this ordering.

/etc/systemd/system/fulcrum.service

[Service]

[Install]

⏱
Time-Gate 2 — Starts Now
Once you start Fulcrum after IBD is complete, it begins building its address index. This takes 4–20 hours. Port 50001 will be closed and refuse connections until indexing is 100% complete. This is normal behavior. Monitor progress with journalctl -u fulcrum -f.
bash — only run after IBD is confirmed complete
systemctl daemon-reload
systemctl enable fulcrum
systemctl start fulcrum

# Monitor indexing progress
journalctl -u fulcrum -f
# Look for "Synch: ... height=..." lines counting up to the chain tip.
# When it says "Fulcrum is up and running" the index is complete.

Verify Fulcrum is listening (after indexing completes)

bash
ss -tlnp | grep 50001
# Should show:  127.0.0.1:50001  LISTEN
05 — Desktop Environment

Install XFCE

⏱
Install XFCE After Both Time-Gates Complete
Do not install XFCE until both IBD and Fulcrum indexing are confirmed complete. Both processes benefit from maximum available RAM. XFCE at idle consumes 300–500 MB — a meaningful fraction during resource-intensive sync. Once Fulcrum reports "Fulcrum is up and running" and is in steady-state, the RAM pressure is low enough that a desktop adds no practical cost.

XFCE is the right choice here: it is lightweight, stable, ships in the official Debian repositories, and installs no background sync daemons or telemetry services. Install only the core desktop — not the full metapackage with bundled applications.

bash
apt update
apt install -y xfce4 xfce4-terminal lightdm

# Install a minimal set of useful utilities
apt install -y thunar mousepad ristretto xfce4-screensaver

# Do NOT install the full xfce4-goodies metapackage
# It pulls in many extras you don't need

Boot Configuration — Text Login by Default

Configure the system to boot to a text login prompt rather than automatically launching the display manager. Since you use this machine infrequently, there is no reason to run a graphical session when you are not actively using Sparrow. Bitcoind and Fulcrum will start immediately on boot without waiting for a desktop session, and the system uses less power at idle.

bash
# Set default boot target to multi-user (text login)
systemctl set-default multi-user.target

# Disable LightDM from starting automatically on boot
systemctl disable lightdm

# To start the desktop when you need it:
systemctl start lightdm
# Or, from a text login, simply run:
startx
Screen Lock
When using the desktop, ensure the screen locks on inactivity and on lid close. XFCE Screensaver handles this. Go to Applications → Settings → Screensaver and enable locking. Set a short timeout — 5 minutes is reasonable. LUKS protects you if the laptop is stolen powered-off; the screen lock protects you when it is sleeping in a bag.

Connect Sparrow to Your Fulcrum Server

Because Sparrow and Fulcrum are on the same machine, the connection is entirely local. No Tor proxy, no .onion address, no SSL — just a direct loopback connection that never leaves the laptop.

Configure Sparrow server connection

Open Sparrow Wallet from the XFCE application menu, then go to File → Preferences → Server:

Sparrow Server Settings
Server Type:     Private Electrum Server
URL:             127.0.0.1:50001
Use SSL:         Off   (local loopback — no encryption needed)
Use Proxy:       Off   (no proxy — Sparrow and Fulcrum are on the same machine)

Click Test Connection. If Fulcrum has finished indexing, you will see a green checkmark and the server's current block height. If the connection fails, verify:

  • Fulcrum indexing is complete (journalctl -u fulcrum | tail -20)
  • Fulcrum is listening (ss -tlnp | grep 50001 — should show 127.0.0.1:50001 LISTEN)
  • The Fulcrum service is running (systemctl status fulcrum)
  • bitcoind is running and fully synced (systemctl status bitcoind)
Setup Complete
Once the test connection succeeds, Sparrow will synchronize your wallet against your own node. Your wallet queries never leave the machine. Bitcoin Core's peer connections route through Tor, keeping your IP off the P2P network. No third party ever sees your addresses, balances, or transaction history.
Appendix A

Maintenance & Upgrades

Routine tasks for keeping your sovereignty stack healthy.

Monitoring Commands

bash — useful daily commands
# Bitcoin Core status
systemctl status bitcoind
sudo -u bitcoin bitcoin-cli -conf=/var/lib/bitcoin/.bitcoin/bitcoin.conf getblockchaininfo

# Fulcrum status
systemctl status fulcrum
journalctl -u fulcrum --since "1 hour ago"

# Tor status
systemctl status tor

# Disk usage
df -h
du -sh /var/lib/bitcoin/.bitcoin/blocks
du -sh /var/lib/fulcrum/db

# Fulcrum admin info (requires python3)
sudo -u fulcrum FulcrumAdmin -p 8000 getinfo

Upgrading Bitcoin Core

bash
# 1. Stop the service gracefully (allows clean shutdown)
systemctl stop bitcoind

# 2. Download and verify the new version (same process as initial install)
#    (See Section 02 — substitute the new version number)

# 3. Install new binaries
install -m 0755 /tmp/bitcoin-NEW_VERSION/bin/bitcoind /usr/local/bin/
install -m 0755 /tmp/bitcoin-NEW_VERSION/bin/bitcoin-cli /usr/local/bin/

# 4. Restart
systemctl start bitcoind

Upgrading Fulcrum

bash
# Stop Fulcrum first (it needs a clean shutdown to avoid DB corruption)
systemctl stop fulcrum

# Verify and install new binary (same process as initial install)
install -m 0755 /tmp/Fulcrum-NEW_VERSION-x86_64-linux/Fulcrum /usr/local/bin/Fulcrum

# Restart (no re-indexing needed for minor version upgrades)
systemctl start fulcrum

LUKS Header Backup

Back up your LUKS header to an offline location (USB drive stored separately from the machine). If the header is corrupted — even if you know the passphrase — you cannot recover the data.

bash — identify your encrypted partition first
lsblk
# Identify the LUKS partition (typically /dev/sda2 or /dev/nvme0n1p2)

cryptsetup luksHeaderBackup /dev/sda2 \
  --header-backup-file /path/to/external/luks-header-backup.bin
# Store this file offline. Treat it with the same security as your passphrase.

Check for Software Updates

bash
# System packages (auto-upgrades handle security, but check manually too)
apt update && apt list --upgradable

# Check current Bitcoin Core version vs latest
bitcoind --version
# Compare against: https://bitcoincore.org/en/releases/

# Check current Fulcrum version vs latest
Fulcrum --version
# Compare against: https://github.com/cculianu/Fulcrum/releases
